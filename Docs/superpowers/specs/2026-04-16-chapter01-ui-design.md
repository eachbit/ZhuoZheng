# 第一章正式 UI 设计文档

日期：2026-04-16  
项目：ZhuoZheng / Garden_Main  
范围：将当前第一章临时运行时 UI 替换为符合既定美术方向的正式 UI。

## 一、目标

在现有 `Garden_Main` 场景中完成第一章整套正式 UI，且**不改变第一章玩法规则**。最终结果需要满足：

- 替换当前由 `PrototypeRuntimeUI` 通过 `OnGUI` 绘制的临时 UI
- 保留现有第一章流程、存档行为和交互规则
- 遵循已经确定的视觉方向：`园亭匾额 + 深墨金边`
- 便于后续同学继续替换美术外观，而不需要重写玩法逻辑

这次工作属于 **UI 替换与 UI 结构重建**，不是玩法重做。  
第一章的暗闸校准、水流方向选择、残页拾取、对话流程、提示弹条等玩法逻辑，功能上保持不变。

## 二、当前现状

当前项目通过 `PrototypeRuntimeUI` 使用 `OnGUI` 绘制第一章和第二章的临时界面。  
这个临时层已经提供了第一章玩法所需的基础展示能力，主要包括：

- 残页计数
- 目标文本
- 顶部提示 / 结果提示
- 交互提示
- 对话框
- 水流方向选择面板
- 第二章答题面板

第一章玩法流程由 `Chapter01Director` 主导。  
它目前已经决定了以下内容：

- 什么时候切换目标文本
- 玩家什么时候可以开始校准暗闸
- 什么时候打开水流选择界面
- 水流选项显示什么文字
- 什么时候显示结果提示
- 什么时候允许拾取残页

因为这些规则已经比较稳定，所以正式 UI 应作为**表现层**接入，而不是再去改动第一章玩法判断。

## 三、已确认的视觉方向

主方向：`园亭匾额 + 深墨金边`

整体视觉语言如下：

- 面板底色使用偏墨绿色的深色，而不是纯黑
- 边框使用细金边，不做厚重描边和夸张浮雕
- 标题区域带有园林匾额、题景框的感觉
- 文本使用米白色，保证可读性
- 标题要雅，正文要清楚
- 不使用明显的西幻风厚重装饰
- 不使用高饱和霓虹色或过量花纹

字体系统已经统一为：

- 标题 / 面板标题：`SourceHanSerifSC-SemiBold.otf`
- 正文 / 选项 / 提示：`SourceHanSansSC-Medium.otf`
- 数字 / 强调信息：`SourceHanSansSC-Bold.otf`

## 四、第一章需要制作的 UI 模块

第一章正式 UI 这次一共包含以下 7 块：

1. 残页计数
2. 目标栏
3. 顶部提示 / 结果提示条
4. 底部交互提示
5. 对话框
6. 水流方向选择面板
7. 暗闸校准面板

第二章答题 UI 本次**不在范围内**。

## 五、各模块设计说明

### 1. 残页计数

作用：  
显示当前已经收集的残页数量。

表现形式：

- 放在左上角
- 面板较小
- 深墨底 + 金边
- 数字重点突出

需要的数据：

- 当前残页数
- 总残页数

数据来源：

- `PrototypeRuntimeUI.SetPageCount`

### 2. 目标栏

作用：  
显示当前第一章主目标。

表现形式：

- 位于左上角，残页计数下方
- 采用窄长牌匾式或简洁任务板式
- 如有小标题可使用宋体
- 正文使用黑体
- 宽度尽量稳定，避免文本切换时界面跳动过大

需要的数据：

- 当前目标文本

数据来源：

- `GardenGameManager.SetObjective`

### 3. 顶部提示 / 结果提示条

作用：  
显示短时反馈，例如水流方向是否正确。

表现形式：

- 位于顶部中间
- 深墨面板 + 金边
- 支持正确 / 错误状态下的颜色强调
- 信息层级要明显强于临时版本

需要的数据：

- 提示文字
- 结果标题
- 结果说明
- 强调色

数据来源：

- `ShowToast`
- `ShowDirectionResult`

### 4. 底部交互提示

作用：  
在玩家可交互时显示简短提示。

表现形式：

- 位于底部中间
- 比对话框小很多
- 样式克制，不能抢主画面

需要的数据：

- 当前交互提示文本

显示规则：

- 当对话、水流选择、暗闸校准、第二章答题等面板开启时隐藏

### 5. 对话框

作用：  
显示叙事对话和引导文字。

表现形式：

- 位于底部，宽版面板
- 带小型人物名牌 / 题签结构
- 说话人名字用宋体
- 正文用黑体
- 保留继续按钮和继续提示

需要的数据：

- 说话人名字
- 对话正文
- 继续回调

数据来源：

- `ShowDialogue`
- `AdvanceDialogue`

### 6. 水流方向选择面板

作用：  
在玩家选择水流方向时作为第一章的核心模态面板。

表现形式：

- 位于画面中央
- 是整套 UI 中完成度最高、最正式的一块
- 顶部有明显题签 / 匾额式标题区
- 正文说明清楚
- 选项按钮规整，带编号
- 保留键盘数字快捷键提示

需要的数据：

- 标题
- 选项文本
- 当前选择回调

行为要求：

- 保留当前数字键选择逻辑
- 保留关闭 / 取消能力

### 7. 暗闸校准面板

作用：  
在玩家旋转暗闸时，提供正式 UI 反馈。

表现形式：

- 位于右侧
- 独立面板
- 边框感更强
- 角度信息、有效范围、操作提示要清晰

需要从玩法层补充的数据：

- 当前是否处于暗闸校准状态
- 当前角度 / 校准显示值
- 正确范围或“可确认”状态
- 当前是左闸还是右闸（如果可区分）

重要说明：

当前临时 `OnGUI` 版本里并没有真正独立的正式“暗闸校准面板”。  
这意味着本次正式 UI 实现，不只是换皮，还需要**新增暗闸校准的专用表现层**。

## 六、技术实现方向

推荐方案：

- 新建基于 `Canvas` 的第一章正式运行时 UI
- `PrototypeRuntimeUI` 保留为兼容层，或在新 UI 接通后逐步退出第一章表现职责
- 尽量保持现有玩法调用接口不变

整体结构建议：

1. 在 `Garden_Main` 原场景中建立 `Chapter01UIRoot`
2. 将每个 UI 模块拆成独立子对象 / 子面板
3. 让现有运行时调用转向新的正式 UI 表现层
4. 为暗闸校准单独增加右侧面板的数据更新通路
5. 将字体、颜色、边框、贴图等整理成便于同学换皮的结构

## 七、为了方便同学后续修改 UI 的结构要求

这套 UI 必须为后续“换边框、换字体、换贴图”预留空间。

规则如下：

- 背景图、边框图、装饰图尽量做成可序列化字段
- 颜色尽量集中管理，不要在很多地方写死
- 数据更新逻辑与样式设置逻辑分开
- 层级命名按功能命名，而不是按美术临时名字命名

建议层级：

- `Chapter01UIRoot`
- `TopLeft/PageCounterPanel`
- `TopLeft/ObjectivePanel`
- `TopCenter/ToastPanel`
- `BottomCenter/InteractionPromptPanel`
- `Bottom/DialoguePanel`
- `Center/FlowChoicePanel`
- `Right/GateCalibrationPanel`

这样后续同学主要就可以替换：

- 边框图
- 背景贴图
- 装饰元素
- 字体资源
- 间距与布局细节

而不用去碰第一章玩法逻辑。

## 八、交互与数据流

玩法归属保持不变：

- `Chapter01Director` 继续负责第一章状态切换与正确性判断
- `GardenGameManager` 继续负责高层 UI 调用
- 新的第一章正式 UI 只负责表现

数据流应为：

1. 第一章玩法状态变化
2. `GardenGameManager` 或 `Chapter01Director` 发出 UI 调用
3. 第一章正式 UI 更新对应面板
4. 模态面板通过已有回调把选择结果回传给玩法层

对于暗闸校准，由于现在没有正式独立面板，所以这部分需要新增专门的数据传递路径。

## 九、容错与安全要求

如果某个 UI 面板缺失：

- 不应阻断第一章玩法
- 尽量静默失败
- 不应因为某个面板没挂好就让回调报错

如果字体资源缺失：

- 必须保留可读性
- 不能出现整块文本消失

如果正式美术资源还没齐：

- 也要能先用简化边框和配色把结构跑起来

## 十、验收测试要求

在认为第一章 UI 完成之前，至少验证以下内容：

1. 开场结束后，目标栏能正常出现
2. 与暗闸交互时，校准面板能正常显示和关闭
3. 左右暗闸都完成后，水流选择面板能正常弹出
4. 选择错误方向后，顶部结果提示和目标文本都会更新
5. 选择正确方向后，残页可拾取，反馈可读
6. 对话框在明亮和较暗场景下都清楚
7. 数字键快捷选择仍然有效
8. 第一章正式 UI 不应破坏第二章逻辑

## 十一、本次不包含的内容

以下内容不属于这次第一章 UI 制作范围：

- 第二章正式 UI
- 新玩法规则
- 重新搭场景
- 纯演出级镜头包装
- 配音表现
- 整体美术管线重构

## 十二、推荐制作顺序

建议按这个顺序实施：

1. 先搭 Canvas 根节点和共享样式
2. 先做左上角 HUD（残页计数、目标栏）
3. 做对话框
4. 做顶部提示 / 结果提示
5. 做底部交互提示
6. 做水流方向选择面板
7. 做暗闸校准面板
8. 最后联调整个第一章流程

## 十三、最终建议

建议采用 **Canvas 化第一章正式 UI** 的实现方式，并保持现有玩法 API 稳定。  
这是目前在以下几项之间最平衡的方案：

- 视觉完成度
- 实现可控性
- 便于同学后续换皮
- 降低玩法回归风险
